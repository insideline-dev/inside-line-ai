## 2. Today's blockers and how I got past them

### 2.1 `git add -p` non-interactive failure (carried over)
Same as yesterday — committed in file-group splits rather than per-line.

### 2.2 Triage v1→v2 reason-code collision
When a lens with `signal=advance` was synthetically downgraded to review for low evidence, the policy was emitting BOTH `lens.X.review` AND `lens.X.low_evidence`. Real ambiguity — the lens didn't actually say review, the downgrade is policy-driven. Fix: suppress `lens.X.review` when the same lens key is in `lowEvidenceKeys`. Reason code is unambiguous now.

### 2.3 Advisor caught a thesis-fit data-flow gap I'd shipped over
For DS-E4-F1-S1, I wired the screening processor to read `MAX(thesisFitScore)` from `startupMatch` rows. Advisor pointed out: SCREENING runs *before* matching in the pipeline (`pipeline.config.ts:90-101` — SCREENING depends on RESEARCH, runs parallel with EVALUATION; matching is post-pipeline). So `maxThesisFitScore` returns null on every first run and the gate effectively never fires for new deals — only on **re-screens** after matching populates rows. Decision: ship the policy code (it's correct), document the gap inline with the path forward (post-matching re-decide hook OR move match calc into the pipeline). Tracked.

### 2.4 Real production bugs surfaced by E2E
The deferred real-data E2E from yesterday surfaced two bugs in already-shipped DS-E3:
- **OpenAI strict-output rejection**: `websiteUrl: z.string().url().optional()` needs `.nullable()`. Every fund-site scrape was failing. Switched to `.url().nullish()`. Critical — the entire onboarding feature was broken in prod.
- **Banner stuck after `investor.onboarding.failed`**: backend deliberately preserves `websiteScrapedAt` for audit; frontend used it to derive "in-flight". Banner span forever. Added a local `hasRecentFailure` flag toggled by the WS handler.

Both shipped in `e599ddb`.

### 2.5 Advisor caught false-positive vectors in dealbreaker rules (DS-E4-F3-S1)
Initial draft used naked substring on a joined haystack (description included), so `"adult"` matched "young adult fiction" and `"ai"` matched "training". Industry rule was strict equality, so `"B2B SaaS"` vs `"SaaS"` triggered hard violations. Fixes:
- Word-boundary regex for explicit dealbreakers, fielded values only (industry/sector), description excluded.
- Bidirectional fuzzy match for industry rule (matches geography rule's behavior).
- `industry_not_in_scope` downgraded to `soft` severity since strict equality on free-text is too brittle.
- Thesis query gated by `enabled: !!startupId` so non-investor render paths don't 404-spam.

### 2.6 Deferred system-level dealbreaker enforcement (DS-E4-F3-S1)
Story implies BOTH per-investor surface AND system-level (so the lens prompts don't trust the LLM to interpret narrative dealbreakers). Built per-investor only. Deferred system-level because it requires aggregating across investors at screening time (same complexity class as the DS-E4-F1-S1 thesis-fit issue). Acknowledged risk: the LLM-generated memo body might still gush about a deal the dealbreaker banner says "doesn't fit". Track the dissonance.

### 2.7 Admin couldn't trigger screening to validate the deal-card
E2E couldn't capture the deal card with real triage data because no startup had been screened — and the lens framework shipped today, so no historical screening rows. There was no admin UI to manually kick off SCREENING (only the full-pipeline approve flow). Added `AdminScreeningService.triggerScreeningForStartup` + `POST /admin/startups/:id/screen` + a "Re-run screening" button on the admin sidebar. Reuses the latest `pipelineRunId` so lens rows attach to a real run.
